# Тестирование QuizLive на группе пользователей

Документ описывает сценарий приёмочного тестирования (п. 6 ТЗ) для MVP платформы квизов.

## Подготовка

### Требования

- Node.js 20+
- 1 компьютер-ведущий (организатор)
- 3–10 устройств участников (ноутбуки, телефоны, планшеты)
- Все устройства в одной сети **или** доступ к серверу по IP/домену

### Запуск

```bash
cd quiz-platform
cp .env.example .env
npm install
npm run db:migrate
npm run db:seed      # демо-аккаунты и готовый квиз
npm run dev
```

Откройте **http://localhost:3000** (на телефонах в той же Wi‑Fi сети: `http://<IP-компьютера>:3000`).

### Демо-аккаунты (после `npm run db:seed`)

| Роль | Email | Пароль |
|------|-------|--------|
| Организатор | `organizer@demo.local` | `demo1234` |
| Участник 1–5 | `player1@demo.local` … `player5@demo.local` | `demo1234` |

## Состав тестовой группы

| Роль | Кол-во | Задача |
|------|--------|--------|
| Организатор | 1 | Создание/запуск квиза, управление вопросами |
| Участники | 3–10 | Подключение по коду, ответы на вопросы |
| Наблюдатель (опц.) | 1 | Фиксация багов, таймингов, UX |

## Сценарий тестирования

### Этап 1 — Регистрация и авторизация (п. 3)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 1.1 | Организатор: `/login` → вход `organizer@demo.local` | Редирект на `/dashboard` | |
| 1.2 | Участник: `/register` → новый аккаунт с ролью «Участник» | Успешная регистрация | |
| 1.3 | Неверный пароль | Сообщение об ошибке | |
| 1.4 | Выход и повторный вход | Сессия восстанавливается | |

### Этап 2 — Создание квиза (п. 3–4)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 2.1 | Дашборд → «Создать квиз» или использовать демо-квиз | Квиз отображается в списке | |
| 2.2 | Редактор → добавить вопрос SINGLE + TEXT | Вопрос сохранён | |
| 2.3 | Добавить вопрос MULTIPLE | Несколько правильных ответов | |
| 2.4 | Добавить вопрос с IMAGE (URL картинки) | Превью изображения | |
| 2.5 | Нажать «Запустить» | Создана сессия, показан код комнаты | |

### Этап 3 — Подключение участников (п. 5 WebSocket)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 3.1 | Участники: `/join` → ввод кода комнаты | Переход на `/play/[code]` | |
| 3.2 | Организатор видит счётчик участников | Число растёт в реальном времени | |
| 3.3 | Участник видит «Ожидайте начала…» | Состояние LOBBY | |

### Этап 4 — Проведение квиза в реальном времени (п. 5)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 4.1 | Организатор: «Начать квиз» | У всех появился 1-й вопрос + таймер | |
| 4.2 | Участник отвечает до истечения таймера | «Ответ принят», начислены баллы | |
| 4.3 | Участник пытается ответить после таймера | Ответ не принимается | |
| 4.4 | Организатор: «Показать ответ» | Подсветка правильных вариантов | |
| 4.5 | Организатор: «Следующий вопрос» | Новый вопрос, сброс выбора | |
| 4.6 | Повторить 4.1–4.5 для всех вопросов | Синхронизация у всех клиентов | |

### Этап 5 — Подсчёт баллов и победители (п. 4–5)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 5.1 | Быстрый правильный ответ (режим SPEED) | Больше баллов, чем у медленного | |
| 5.2 | Неправильный ответ | 0 баллов | |
| 5.3 | После последнего вопроса → «Следующий» | Экран FINISHED, лидерборд | |
| 5.4 | Топ-3 отображается корректно | Порядок по `totalScore` | |

### Этап 6 — Личный кабинет (п. 3–4)

| # | Действие | Ожидаемый результат | ✓ |
|---|----------|---------------------|---|
| 6.1 | Участник: `/profile` | История с баллами и названием квиза | |
| 6.2 | Организатор: `/profile` | История проведённых сессий | |

## Нагрузочный мини-тест

1. Подключите **5+ участников** одновременно.
2. Все отвечают на один вопрос в течение 5 секунд.
3. Проверьте: нет дублей ответов, лидерборд обновляется, сервер не падает.

---

## Известные ограничения MVP

- Активные комнаты сбрасываются при перезапуске сервера (runtime в памяти).
- Изображения — только по URL (без загрузки файлов).
- SQLite — для production рекомендуется PostgreSQL.

