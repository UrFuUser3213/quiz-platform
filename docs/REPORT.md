## 1. Цель и постановка задачи

Целью работы является разработка работоспособного веб-приложения, позволяющего организатору создавать и проводить квизы, а участникам — подключаться к текущей сессии по коду комнаты, отвечать на вопросы синхронно и получать итоговый рейтинг.

**Функциональные требования и степень выполнения:**

| Требование | Реализация |
|------------|------------|
| Регистрация и авторизация (организатор / участник) | Роли `ORGANIZER` и `PARTICIPANT`, JWT в httpOnly-cookie |
| Создание и настройка квизов | Дашборд, редактор, категории, таймер, режим баллов |
| Вопросы: текст и изображение; одиночный и множественный выбор | `MediaType`, `QuestionType`, редактор с шаблонами |
| Live-сессия по коду комнаты | `QuizSession.roomCode`, Socket.IO |
| Ответ только во время показа вопроса | Проверка `state === QUESTION` и таймера на сервере |
| Подсчёт баллов и определение победителей | Режимы `SPEED` и `CORRECT_ONLY`, лидерборд |
| Личный кабинет с историей | `/profile`, API `/api/profile/history` |

Дополнительно реализованы: каталог квизов (`/explore`), режим тренировки (`/practice`), реакции участников, статистика сессий, дублирование и удаление квизов.

## 2. Обоснование выбора средств разработки

### 2.1. Клиентская часть: HTML, CSS, JavaScript, фреймворк React (Next.js 15)

Приложение построено на **Next.js 15** с **React 19** и **TypeScript**. Next.js выбран как современный full-stack-фреймворк на JavaScript/TypeScript:

- **Единая кодовая база** — страницы UI и REST API находятся в одном репозитории, что упрощает разработку и сопровождение учебного MVP.
- **App Router** — файловая маршрутизация (`/dashboard`, `/host/[code]`, `/play/[code]`) соответствует спроектированным экранам.
- **TypeScript** — статическая типизация снижает количество ошибок при работе с событиями WebSocket и моделями Prisma.
- **Tailwind CSS** — утилитарные классы ускоряют вёрстку; стили соответствуют токенам из макетов Figma (violet-600, slate-900 и т.д.).
- **HTML/CSS** — разметка формируется React-компонентами; глобальные стили — в `globals.css`.

Альтернативы (чистый React + Vite, Vue, Angular) отклонены в пользу Next.js из-за встроенной поддержки API Routes и удобства развёртывания.

### 2.2. Серверная часть: Node.js (JavaScript/TypeScript)

Сервер реализован на **Node.js**:

- **Custom server** (`server.ts`) — HTTP-сервер на базе `http` + Next.js request handler.
- **API Routes** (`src/app/api/*`) — REST-эндпоинты для авторизации, CRUD квизов, сессий, профиля, каталога.
- **Единый язык** — TypeScript на клиенте и сервере снижает порог входа и упрощает передачу типов (например, `RoomSnapshot`).

Выбор Node.js обоснован требованием ТЗ (серверная часть на JavaScript) и нативной интеграцией с Socket.IO.

### 2.3. Взаимодействие с базой данных: Prisma ORM + SQLite

| Критерий | SQLite + Prisma |
|----------|-----------------|
| Установка | Файл `dev.db`, без отдельного DB-сервера |
| Миграции | `prisma migrate dev` — версионирование схемы |
| Типобезопасность | Автогенерация клиента в `src/generated/prisma` |
| Переносимость | Схема совместима с PostgreSQL для production |

**Основные сущности:** `User`, `Quiz`, `Question`, `AnswerOption`, `QuizSession`, `QuizParticipant`, `SessionAnswer`. Связи отражают ER-модель из FigJam (см. приложение).

### 2.4. Realtime: Socket.IO (WebSocket)

Для синхронизации live-квиза выбран **Socket.IO 4**:

- Транспорт WebSocket с fallback на long-polling.
- Примитив **комнат** (`join_room`) соответствует кодам сессий.
- События: `start_quiz`, `submit_answer`, `reveal_answer`, `next_question`, `send_reaction`, `skip_question`, `end_quiz_early`.

Альтернатива (чистый WebSocket API) потребовала бы больше boilerplate для переподключений и комнат.

### 2.5. Проектирование интерфейса: Figma / FigJam

Макеты и user flow разработаны в **Figma Design** и **FigJam** (функциональный аналог Miro для диаграмм). Спецификация экранов — `docs/design/SCREENS.md`, сценарии — `docs/design/USER_FLOW.md`.

### 2.6. Вспомогательные библиотеки

- **bcryptjs** — хеширование паролей  
- **jsonwebtoken** — сессионные токены  
- **zod** — валидация входных данных API  
- **nanoid** — генерация кодов комнат

## 3. Архитектура приложения

```
┌─────────────────────────────────────────────────────────┐
│                    Браузер (клиент)                      │
│  React-страницы + useQuizSocket (Socket.IO client)      │
└───────────────┬─────────────────────┬───────────────────┘
                │ REST /api/*         │ WS /api/socketio
                ▼                     ▼
┌───────────────────────────────────────────────────────────┐
│              Node.js custom server (server.ts)             │
│  ┌─────────────────┐    ┌──────────────────────────────┐  │
│  │ Next.js handler │    │ Socket.IO (src/server/socket) │  │
│  └────────┬────────┘    └──────────────┬───────────────┘  │
│           │                            │                   │
│           └────────────┬───────────────┘                   │
│                        ▼                                   │
│                   Prisma Client                            │
└────────────────────────┬──────────────────────────────────┘
                         ▼
                    SQLite (dev.db)
```

**Жизненный цикл live-сессии:** `LOBBY` → `QUESTION` → `REVEAL` → … → `FINISHED`.

Сервер хранит runtime-состояние комнаты в памяти (Map) и персистирует результаты в БД. Ответ участника принимается только при активном вопросе и неистёкшем таймере.

**Подсчёт баллов:**
- `CORRECT_ONLY` — фиксированные баллы за правильный ответ;
- `SPEED` — баллы пропорциональны оставшемуся времени.

## 4. Основные этапы разработки

| № | Этап | Содержание работ | Результат |
|---|------|------------------|-----------|
| **1** | Проектирование UI | User flow организатора и участника; спецификация 12+ экранов; дизайн-токены; макеты в Figma/FigJam | `docs/design/`, ссылки в приложении |
| **2** | Модель БД | Анализ предметной области; ER-диаграмма; Prisma schema; миграции | `prisma/schema.prisma`, FigJam ERD |

## 5. Тестирование

Проведено функциональное тестирование по чеклисту (`docs/TESTING.md`):

1. Регистрация организатора и участников, вход/выход.
2. Создание квиза с вопросами разных типов, запуск сессии.
3. Подключение 3–5 участников по коду комнаты в разных браузерах.
4. Синхронный показ вопросов, отправка ответов, блокировка после таймера.
5. Лидерборд, история в личном кабинете, статистика сессии.

Демо-аккаунты: `organizer@demo.local` / `demo1234`, `player1@demo.local` … `player5@demo.local` / `demo1234` (создаются командой `npm run db:seed`).

## 6. Ограничения MVP и перспективы

- Изображения подключаются по URL (без загрузки файлов на сервер).
- SQLite — для учебного стенда; для production рекомендуется PostgreSQL.
- Runtime комнат в памяти — при перезапуске сервера активные сессии сбрасываются.
- Нет OAuth и восстановления пароля.

**Возможное развитие:** Redis для сессий, деплой на Vercel/Railway, drag-and-drop вопросов, экспорт результатов в CSV, PWA для мобильных участников.
